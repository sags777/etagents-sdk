# Persistence

Persistence lets the SDK write completed run artifacts and HITL checkpoints to a `StoreProvider`. Attach a store when you want durable audit trails, resumable approvals, or offline inspection of run data.

```typescript
import { createAgent, startRun, FileStore } from "@etagents/sdk";

const agent = createAgent({
  name: "assistant",
  systemPrompt: "You are a helpful assistant.",
  model: "claude-sonnet-4-6",
  store: new FileStore(".sessions"),
});

const result = await startRun(agent, "Summarise the latest support ticket");
console.log(result.status);
```

New runs always get a fresh kernel-generated `runId`. Caller-supplied `RunConfig.runId` is deprecated and ignored for new runs. Durable continuation is provided by `continueRun(checkpointId, decisions, { agent })` for HITL checkpoints, not by reusing `runId` on `startRun()`.

---

## FileStore

`FileStore` writes one JSON file per key to a local directory. No external dependencies required.

```typescript
import { FileStore } from "@etagents/sdk";

const store = new FileStore(".sessions");
```

Keys are mapped to files under `{baseDir}/{key}.json`. Use `FileStore` for:
- Local development
- Single-process deployments
- Testing

:::warning
`FileStore` is not safe for concurrent access to the same key from multiple processes. Use `RedisStore` for multi-instance deployments.
:::

## RedisStore

`RedisStore` stores sessions in Redis. Use this for multi-instance deployments, serverless environments, or anywhere you need TTL-based expiry.

```typescript
import { RedisStore, createRedisStore } from "@etagents/sdk";

const store = await RedisStore.connect({
  url: process.env.REDIS_URL,
  namespace: "app",
});

const lazyStore = createRedisStore({
  url: process.env.REDIS_URL,
  namespace: "app",
});

const agent = createAgent({
  name: "assistant",
  systemPrompt: "...",
  store,
});
```

Use `RedisStore.connect()` in long-lived processes when you want to own the connection lifecycle. Use `createRedisStore()` for sync module init contexts like Next.js route files.

## Session snapshot format

The kernel reconstructs wire-safe `SessionSnapshot` and `SuspendSnapshot` shapes from the normalized store records when needed:

```typescript
interface SessionSnapshot {
  version: 1;
  runId: string;
  messages: Message[];                  // full conversation history
  metadata: Record<string, unknown>;    // arbitrary key-value from RunConfig.metadata
  createdAt: string;                    // ISO-8601
  updatedAt: string;
  insights: {
    facts: string[];
    userFacts: string[];
    summary: string;
    topics: string[];
  };
  _kernel: {
    privacyMap?: PrivacyMap;
    tokenUsage?: TokenUsage;
    configFingerprint?: string;
  };
}
```

The default persistence adapter stores normalized entity families for runs, messages, tool calls, events, checkpoints, and approvals rather than a single opaque blob.

## Metadata

Attach arbitrary metadata via `RunConfig.metadata`. It is persisted with run and checkpoint records and threaded into `ToolContext`:

```typescript
await startRun(agent, "Hello", {
  metadata: { userId: "u_123", tier: "pro", source: "web" },
});

// The metadata is written alongside persisted run artifacts and checkpoint snapshots
```

## Stored history limits

Long sessions can exceed the model's context window. The kernel applies internal defaults from `DEFAULT_CONFIG` when persisting large histories:

| Runtime default | Value | Description |
|---|---|---|
| `maxPersistedMessages` | `40` | Older history is compacted before persistence |
| `maxToolContentLength` | `8_000` | Tool result content is truncated before it is appended |

These are runtime defaults, not `AgentConfig` fields. Message trimming always preserves the most recent history and the system prompt.

## HITL checkpoints

When HITL is enabled, the kernel writes a `SuspendSnapshot` checkpoint — a separate record from the session snapshot. HITL checkpoints require a `hitlStore` on the `HitlConfig`:

```typescript
const agent = createAgent({
  name: "assistant",
  systemPrompt: "...",
  store: new FileStore(".sessions"),                  // session history
  hitl: {
    mode: "sensitive",
    hitlStore: createRedisStore({
      url: process.env.REDIS_URL,
      namespace: "hitl",
    }),
  },
});
```

See the [HITL guide](../advanced/hitl) for the full suspend/resume flow.

---

## Error handling

Persistence failures are best-effort. If `store.write()` throws, the error is logged but the run still returns its `RunResult`. A failed persist does not crash the session.

If checkpoint or store reads fail during resume, `continueRun()` surfaces the error to the caller.
