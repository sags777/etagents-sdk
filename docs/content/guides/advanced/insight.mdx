# Insight (Post-Run Reflection)

Insight drives automatic fact extraction after a run. When `insight` is configured, the kernel calls `runInsight()` once the turn loop exits, persists `SessionInsights` in the session snapshot, and indexes typed memory entries through the configured `MemoryProvider`.

```typescript
import { createAgent, startRun, InMemory } from "@etagents/sdk";

const memory = new InMemory();

const agent = createAgent({
  name: "assistant",
  systemPrompt: "You are a helpful assistant.",
  model: "claude-sonnet-4-6",
  memory,
  insight: {
    model: "claude-haiku-4-5",   // use a cheap model for reflection
    maxFacts: 30,
    minTurns: 1,
  },
});

await startRun(agent, "I prefer dark mode and I always use TypeScript.");

// On the next run, extracted facts are injected into context automatically
const result = await startRun(agent, "What are my preferences?");
console.log(result.response);
// "You prefer dark mode and you always use TypeScript."
```

Insight extraction runs after the run exits, on the non-critical path. A failed extraction never crashes the run.

---

## SessionInsights shape

When insight runs, the kernel stores `SessionInsights` in `SessionSnapshot.insights`:

```typescript
interface SessionInsights {
  facts: string[];      // Decisions, preferences, action items
  userFacts: string[];  // Identity facts — name, role, title
  summary: string;      // One-sentence session summary
  topics: string[];     // High-level topic tags
}
```

By default, `facts`, `userFacts`, and `topics` are indexed into memory as `fact`, `user_fact`, and `topic` entries. When `injectSummaryOnly: true`, the SDK indexes only the condensed `summary` as a `summary` entry.

---

## InsightConfig reference

| Field | Type | Default | Description |
|---|---|---|---|
| `model` | `string` | agent's model | Model used for extraction (use a cheaper model) |
| `maxFacts` | `number` | `30` | Cap on session facts (identity facts are separate) |
| `minTurns` | `number` | unset | Skip extraction on short runs below this turn count |
| `hypothesize` | `boolean` | `false` | Generate a hypothetical answer before memory retrieval (HyDE) |
| `injectSummaryOnly` | `boolean` | `false` | Index only `summary` instead of the full fact set |
| `prompts.extractFacts` | `string` | built-in | Override the extraction prompt |
| `prompts.summarise` | `string` | built-in | Override the summary prompt |

```typescript
insight: {
  model: "claude-haiku-4-5",
  maxFacts: 20,
  minTurns: 2,
  prompts: {
    extractFacts: "Extract key decisions, user preferences, and open action items only.",
  },
},
```

## Deduplication

Facts are deduplicated across runs using three tiers:

1. **Exact normalised match** — lowercased, trimmed, whitespace-collapsed. Exact duplicates are dropped.
2. **Levenshtein near-duplicate** — similarity ≥ 0.85. Near-identical phrasings are merged.
3. **Cap with eviction** — when `maxFacts` is reached, task-outcome facts (at the end of the pool) are evicted first. Identity facts (`userFacts`) are never evicted.

This means: the more often a preference appears across sessions, the more likely it is to survive eviction.

---

## Fact categories

**`userFacts`** are identity-stable facts: name, role, organisation, timezone, language. They are capped at 10 and never evicted. The extraction model is prompted to identify them separately.

**`facts`** are session-level observations: decisions made, preferences expressed, tasks completed. They accumulate up to `maxFacts` and are evicted when the cap is reached.

**`topics`** are short labels that make future retrieval easier to route and rank. They are indexed as typed memory entries when `injectSummaryOnly` is false.

---

## Choosing the right model

Insight extraction makes one LLM call per run. A fast, inexpensive model is ideal:

| Model | Quality | Cost | Recommended for |
|---|---|---|---|
| `claude-haiku-4-5` | Good | Low | Most use cases |
| `claude-sonnet-4-6` | Better | Medium | High-value sessions where accuracy matters |

Avoid using the same model for both the agent and insight extraction in production — extraction is a background task that should not compete with the agent's token budget.
