# Error Handling

The SDK uses a typed error hierarchy rooted at `EtaError`. Every error has a `code` string for programmatic handling and a human-readable `message`. Errors surface through `RunResult.status`, thrown exceptions, and `RunEvent` error events.

---

## Error taxonomy

```
EtaError
├── ModelError       — unsupported model shorthand or provider setup failure
├── StoreError       — store provider failures
├── MemoryError      — memory provider failures
├── PrivacyError     — privacy provider failures
├── ToolError        — tool validation or execution failures
├── McpError         — MCP client/server failures
├── BudgetError      — budget helper failures
└── CheckpointError  — HITL resume failures
```

All exported SDK errors extend `EtaError`, which extends `Error`. The `code` is coarse-grained and stable, for example `"TOOL_ERROR"` or `"CHECKPOINT_ERROR"`.

You can catch at any level:

```typescript
import { CheckpointError, EtaError, ModelError } from "@etagents/sdk";

try {
  const result = await continueRun(checkpointId, decisions, { agent });
} catch (err) {
  if (err instanceof CheckpointError) {
    // checkpoint missing or decisions were incomplete
  } else if (err instanceof ModelError) {
    // unsupported string model shorthand during createAgent()
  } else if (err instanceof EtaError) {
    // any SDK error
  } else {
    throw err; // not ours
  }
}
```

## RunResult status vs thrown exceptions

Most run failures surface through `RunResult.status`, not thrown exceptions. Know which is which:

| Condition | How it surfaces |
|---|---|
| Agent completed normally | `status: "complete"` |
| Token budget exceeded | `status: "budget_exceeded"` (no throw) |
| AbortSignal fired | `status: "cancelled"` (no throw) |
| HITL suspension | `status: "awaiting_approval"` (no throw) |
| Unexpected error in turn loop | `status: "error"` + `result.response` contains error message |
| Checkpoint missing or decisions incomplete (`continueRun`) | Throws `CheckpointError` |
| Unsupported string model shorthand (`createAgent`) | Throws `ModelError` |

Only `continueRun` and `createAgent` commonly throw. `startRun` almost never throws — check `status` instead.

---

## ModelError

`createAgent` throws `ModelError` when a string model shorthand does not match a supported provider prefix. All `ModelError` instances use the fixed code `"MODEL_ERROR"`.

```typescript
import { ModelError, createAgent } from "@etagents/sdk";

try {
  createAgent({
    name: "assistant",
    systemPrompt: "You are helpful.",
    model: "bad-model",
  });
} catch (err) {
  if (err instanceof ModelError) {
    console.error(err.code, err.message);
  }
}
```

## ToolError

`ToolError` represents schema-validation and handler failures inside tool execution. In normal runs it does **not** escape from `startRun` — the kernel packages the failure as a tool result, marks it with `isError: true`, and sends it back to the model.

All `ToolError` instances use the fixed code `"TOOL_ERROR"`. The detailed reason stays in the error message and the tool-result payload.

To detect tool errors in your code:

```typescript
const result = await startRun(agent, input, {
  onEvent(event) {
    if (event.kind === "tool_result" && event.isError) {
      console.error(`Tool ${event.toolCallId} failed: ${event.result}`);
    }
  },
});

// Or inspect the toolCalls log
const failedTools = result.toolCalls.filter((t) => t.isError);
```

## CheckpointError

`CheckpointError` is thrown by `continueRun` when the kernel cannot restore a suspended run or when the provided approval decisions do not cover every pending tool call.

All `CheckpointError` instances use the fixed code `"CHECKPOINT_ERROR"`.

```typescript
import { CheckpointError, EtaError } from "@etagents/sdk";

try {
  const result = await continueRun(checkpointId, decisions, { agent });
} catch (err) {
  if (err instanceof CheckpointError) {
    console.error(err.code, err.message);
  }

  if (err instanceof EtaError && err.code === "CHECKPOINT_ERROR") {
    // same condition, if you prefer code-based handling
  }
}
```

## Fail-open subsystems

Several subsystems are fail-open: if they throw, the error is logged and the run continues.

| Subsystem | Failure behaviour |
|---|---|
| Lifecycle hooks | Error logged; run continues |
| Privacy masking | Error logged; run continues with unmasked input |
| Memory indexing | Error logged; run exits normally |
| Session persist | Error logged; `RunResult` still returned |
| Insight extraction | Error logged; run exits normally |

Failures in fail-open subsystems appear as `error` events in `onEvent`. Monitor them in production.

## Production pattern: wrapping startRun

```typescript
import { startRun, EtaError } from "@etagents/sdk";
import type { AgentDef, RunResult } from "@etagents/sdk";

async function runSafe(
  agent: AgentDef,
  input: string,
  runId: string,
): Promise<RunResult | null> {
  try {
    const result = await startRun(agent, input, { runId });

    if (result.status === "error") {
      logger.error("Agent run error", { runId, message: result.response });
      return null;
    }

    if (result.status === "budget_exceeded") {
      logger.warn("Budget exceeded", { runId, turns: result.turns, usage: result.totalUsage });
    }

    return result;
  } catch (err) {
    if (err instanceof EtaError) {
      logger.error("SDK error", { runId, code: err.code, message: err.message });
    } else {
      logger.error("Unexpected error", { runId, err });
    }
    return null;
  }
}
```
