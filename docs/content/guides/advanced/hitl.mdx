# Human-in-the-Loop (HITL)

HITL gates tool calls on human approval. When the agent reaches a tool call requiring approval, the run either suspends and returns `status: "awaiting_approval"`, or continues inline when you use callback mode. In the suspend/resume flow, you present the pending tool calls to a human, collect decisions, then call `continueRun` to resume.

```typescript
import { z } from "zod";
import { createAgent, startRun, continueRun, defineTool, FileStore } from "@etagents/sdk";

const sendEmailTool = defineTool({
  name: "send_email",
  description: "Send an email on behalf of the user.",
  params: z.object({
    to: z.string(),
    subject: z.string(),
    body: z.string(),
  }),
  async handler({ to, subject }) {
    await emailService.send({ to, subject });
    return "Email sent.";
  },
});

const agent = createAgent({
  name: "email-agent",
  systemPrompt: "You draft and send emails on behalf of the user.",
  model: "claude-sonnet-4-6",
  tools: [sendEmailTool],
  hitl: {
    mode: "tool",        // every tool call requires approval
    hitlStore: new FileStore(".checkpoints"),
  },
});

// Step 1: Run suspends when it hits a tool call
const first = await startRun(
  agent,
  "Send a welcome email to alice@example.com",
  { runId: "session-alice" },
);

console.log(first.status); // "awaiting_approval"
console.log(first.pendingApprovals);
// [{ toolCallId: "tc_abc", name: "send_email", args: { to: "alice@example.com", ... }, agentName: "email-agent" }]

// Step 2: Present pendingApprovals to the human and collect decisions
// (your UI / webhook / Slack bot handles this)
const checkpointId = first.checkpointId!;

const decisions = first.pendingApprovals.map((pa) => ({
  toolCallId: pa.toolCallId,
  approved: true,   // or false to deny
}));

// Step 3: Resume
const final = await continueRun(checkpointId, decisions, { agent });
console.log(final.status);    // "complete"
console.log(final.response);  // LLM's final message
```

## HITL modes

| Mode | Behaviour |
|---|---|
| `"none"` | No approval required (default) |
| `"tool"` | Every tool call requires approval |
| `"sensitive"` | Only tools with `sensitive: true` in their `ToolDef` require approval |
| `"callback"` | Calls `onApprove(pending)` inside `startRun()` and continues without persisting a checkpoint |

### sensitive mode

Mark individual tools as sensitive inside `defineTool` and use `mode: "sensitive"` to require approval only for those tools:

```typescript
const sendEmailTool = defineTool({
  name: "send_email",
  sensitive: true,  // this tool requires approval
  // ...
});

const lookupTool = defineTool({
  name: "lookup_contact",
  // sensitive omitted — runs without approval
  // ...
});

const agent = createAgent({
  hitl: { mode: "sensitive", hitlStore: store },
});
```

:::important
**In multi-agent setups each specialist agent must declare its own `hitl` config.** The router's `hitl` setting does not propagate to agents in the pool — each agent runs its own kernel turn loop independently.

```typescript
// ❌ Wrong — only the router has hitl, the agents default to mode:"none"
const coordinator = createAgent({ hitl: { mode: "sensitive" }, ... });
const workerAgent = createAgent({ /* no hitl — never suspends */ ... });

// ✅ Correct — each agent that owns sensitive tools declares its own config
const workerAgent = createAgent({
  hitl: { mode: "sensitive", hitlStore: store },
  tools: [sendEmailTool],
  ...
});
```
:::

## Callback mode

Use callback mode when approvals happen in-process and you do not want to persist checkpoints:

```typescript
const agent = createAgent({
  name: "assistant-inline",
  systemPrompt: "...",
  tools: [sendEmailTool],
  hitl: {
    mode: "callback",
    onApprove: async (pending) => pending.map((approval) => ({
      toolCallId: approval.toolCallId,
      approved: true,
    })),
  },
});

const result = await startRun(agent, "Send a welcome email to alice@example.com");
```

## ApprovalDecision

Each decision object approves or denies a single tool call:

```typescript
interface ApprovalDecision {
  toolCallId: string;
  approved: boolean;
}
```

You can approve some and deny others in the same batch:

```typescript
const decisions = [
  { toolCallId: "tc_abc", approved: true },   // allow
  { toolCallId: "tc_xyz", approved: false },  // deny
];
```

Denied tool calls receive a synthetic result: `"Tool call rejected by human reviewer."` The LLM incorporates this into its final response.

## Suspend/resume in stateless HTTP

HITL is designed for stateless environments where the process may not be alive when the human responds. The `SuspendSnapshot` checkpoint is written to `hitlStore` and can be loaded in any process.

```typescript title="app/api/agent/route.ts"
import { createAgent, startRun, continueRun, createRedisStore } from "@etagents/sdk";
import type { ApprovalDecision } from "@etagents/sdk";

const hitlStore = createRedisStore({ url: process.env.REDIS_URL, namespace: "hitl" });

const agent = createAgent({
  name: "agent",
  systemPrompt: "...",
  hitl: { mode: "sensitive", hitlStore },
});

// POST /api/agent — new run
export async function POST(req: Request) {
  const { prompt, runId, userId } = await req.json() as { prompt: string; runId: string; userId: string };
  const result = await startRun(agent, prompt, {
    runId,
    metadata: { userId },  // ← stored in checkpoint snapshot
  });
  return Response.json(result);
}

// POST /api/agent/resume — human has reviewed
export async function PATCH(req: Request) {
  const session = await getSession(req);
  const { checkpointId, decisions } = await req.json() as {
    checkpointId: string;
    decisions: ApprovalDecision[];
  };
  const result = await continueRun(checkpointId, decisions, {
    agent,
    metadata: { userId: session.id },  // ← re-supply so tool handlers can read it
  });
  return Response.json(result);
}
```

:::important
**Always re-supply `metadata` in `continueRun`'s `RestoreConfig`.** The checkpoint snapshot stores a copy of the original metadata, but for security-sensitive values such as `userId` it is best practice to re-derive them from the live session on resume rather than trusting the persisted copy.
:::

The `hitlStore` is separate from the session `store` in `AgentConfig`. Use Redis-backed storage for `hitlStore` in multi-instance deployments so any instance can resume the checkpoint.

## Checkpoint errors

If `continueRun` cannot load a checkpoint, or if you provide an incomplete set of decisions, it throws `CheckpointError` with code `"CHECKPOINT_ERROR"`.

```typescript
import { CheckpointError, EtaError } from "@etagents/sdk";

try {
  const result = await continueRun(checkpointId, decisions, { agent });
} catch (err) {
  if (err instanceof CheckpointError || (err instanceof EtaError && err.code === "CHECKPOINT_ERROR")) {
    // The checkpoint is missing, consumed, or the decisions were incomplete.
  }
}
```

## SSE + HITL

For streaming endpoints that need HITL, use `SessionEventStream.resume`:

```typescript
const eventStream = new SessionEventStream(agent);

// In your resume endpoint:
const body = eventStream.resume(checkpointId, decisions);
return new Response(body, { headers: SSE_HEADERS });
```
