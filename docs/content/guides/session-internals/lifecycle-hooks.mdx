# Lifecycle Hooks

Lifecycle hooks let you inject logic at specific moments in a run without modifying the kernel or your tools. Use them for telemetry, audit logging, pre-flight checks, and post-run side effects.

```typescript
import { createAgent } from "@etagents/sdk";

const agent = createAgent({
  name: "assistant",
  systemPrompt: "...",
  model: "claude-sonnet-4-6",
  hooks: {
    beforeRun: async (input, context) => {
      await telemetry.record("run_start", { input, runId: context.runId });
    },
    onTurnStart: async (turn, context) => {
      console.log(`Turn ${turn}: sending to model`);
    },
    onToolResult: async (result, context) => {
      await telemetry.record("tool_result", { id: result.toolCallId, error: result.isError });
    },
    afterRun: async (result, context) => {
      await auditLog.write({ runId: context.runId, status: result.status });
    },
  },
});
```

Hook behavior is intentionally mixed:

- `beforeRun` and `afterRun` are critical-path hooks. If they throw, the error propagates to the caller.
- `onTurnStart`, `onTurnEnd`, and `onToolResult` are best-effort hooks. They are wrapped in `safeHook()`, so errors are swallowed and the run continues.
- `onToolCall` and `beforeComplete` are declared on the public `LifecycleHooks` type but are not invoked by the runtime yet. Use `onEvent` when you need `tool_call` or `complete` notifications today.

---

## Available hooks

| Hook | Signature | Runtime status | When |
|---|---|---|
| `beforeRun` | `(input: string, context: HookContext) => void \| Promise<void>` | wired, errors propagate | Before memory retrieval and the first model turn |
| `afterRun` | `(result: RunResult, context: HookContext) => void \| Promise<void>` | wired, errors propagate | After persistence and run finalization |
| `onTurnStart` | `(turn: number, context: HookContext) => void \| Promise<void>` | wired, fail-open | Before each LLM call |
| `onTurnEnd` | `(turn: number, context: HookContext) => void \| Promise<void>` | wired, fail-open | After each LLM response |
| `onToolResult` | `(result: ToolResult, context: HookContext) => void \| Promise<void>` | wired, fail-open | After each tool returns |
| `onToolCall` | `(call: ToolCall, context: HookContext) => void \| Promise<void>` | declared, not invoked yet | Reserved for future tool-call-start hooks |
| `beforeComplete` | `(messages: Message[], context: HookContext) => void \| Promise<void>` | declared, not invoked yet | Reserved for future final-message inspection |

All hooks may be async. Wired hooks are awaited before the runtime continues.

## beforeRun / afterRun

Use `beforeRun` for pre-flight checks or request-scoped setup that must succeed before the run begins. Use `afterRun` for critical post-run side effects:

```typescript
hooks: {
  beforeRun: async (input, context) => {
    await quota.assertCanRun(context.runId, input);
  },
  afterRun: async (result, context) => {
    await audit.save({ runId: context.runId, status: result.status });
  },
},
```

Because these hooks are on the critical path, errors are **not** swallowed.

---

## onTurnStart / onTurnEnd

Use these to instrument LLM call latency:

```typescript
const turnStartTimes = new Map<number, number>();

const agent = createAgent({
  ...,
  hooks: {
    onTurnStart: async (turn) => {
      turnStartTimes.set(turn, Date.now());
    },
    onTurnEnd: async (turn, context) => {
      const start = turnStartTimes.get(turn);
      if (start) {
        const latencyMs = Date.now() - start;
        await metrics.record("llm_latency", { turn, latencyMs });
      }
    },
  },
});
```

---

## onToolResult

Use this to audit tool outcomes:

```typescript
hooks: {
  onToolResult: async (result, context) => {
    // result.toolCallId, result.result (string), result.isError, result.durationMs
    if (result.isError) {
      await alerting.trigger("tool_error", { id: result.toolCallId });
    }
  },
},
```

## Reserved hook slots

`onToolCall` and `beforeComplete` are part of the public `LifecycleHooks` type, but the current runtime does not invoke them yet. If you need those notifications today:

- use `onEvent` with `tool_call` to observe tool dispatch starts
- use `onEvent` with `complete` to observe run completion

---

## Hooks vs onEvent

| | Hooks | `onEvent` |
|---|---|---|
| Defined on | `AgentConfig` (applies to all runs) | `RunConfig` (per-run) |
| Error handling | Mixed: `beforeRun` / `afterRun` propagate, turn/tool hooks fail-open | Fail-open |
| Access to run context | Yes (`agentName`, `runId`, `turn`) | Depends on the event payload |
| Use for | Persistent telemetry, audit logging | Per-run UI updates, progress display |

For simple per-run observation, prefer `onEvent`. Reserve hooks for cross-cutting concerns you want on every run using this agent.
